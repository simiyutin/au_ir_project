         <div class="magic-block-callout type-info "> 
          <h3><i title="Info" class="fa fa-info-circle"></i>Apache Ignite is ... </h3> 
          <div class="callout-body">
           <p>the <strong>in-memory</strong> computing platform<br>that is <strong>durable</strong>, <strong>strongly consistent</strong>, and <strong>highly available</strong><br>with powerful <strong>SQL</strong>, <strong>key-value</strong> and <strong>processing</strong> APIs</p> 
          </div> 
         </div> 
         <div class="magic-block-image"> 
          <figure>
           <a href="https://files.readme.io/a8fff03-durable-memory.png" class="block-display-image-parent block-display-image-size-smart "><img src="https://files.readme.io/a8fff03-durable-memory.png"></a> 
          </figure> 
         </div> 
         <div class="magic-block-api-header"> 
          <h1 class="header-scroll is-api-header"><span id="durable-memory"></span> 
           <div id="section-durable-memory" class="anchor waypoint"></div>Durable Memory<a href="#section-durable-memory" class="fa fa-anchor"></a> </h1> 
         </div> 
         <div class="magic-block-textarea">
          <p>Ignite's durable memory component treats RAM not just as a caching layer but as a complete fully functional storage layer. This means that users can turn the persistence on and off as needed. If the persistence is off, then Ignite can act as a distributed <strong>in-memory database</strong> or <strong>in-memory data grid</strong>, depending on whether you prefer to use SQL or key-value APIs. If the persistence is turned on, then Ignite becomes a distributed, <strong>horizontally scalable database</strong> that guarantees full data consistency and is resilient to full cluster failures.</p> 
          <p><a href="https://apacheignite-cpp.readme.io/docs/durable-memory" target="_self">Read More</a></p> 
         </div> 
         <div class="magic-block-api-header"> 
          <h1 class="header-scroll is-api-header"><span id="ignite-persistence"></span> 
           <div id="section-ignite-persistence" class="anchor waypoint"></div>Ignite Persistence<a href="#section-ignite-persistence" class="fa fa-anchor"></a> </h1> 
         </div> 
         <div class="magic-block-textarea">
          <p>Ignite Native Persistence is a distributed, ACID, and SQL-compliant disk store that transparently integrates with Ignite's Durable Memory as an optional disk layer storing data and indexes on SSD, Flash, 3D XPoint, and other types of non-volatile storages.</p> 
          <p>With the Ignite Persistence enabled, you no longer need to keep all the data and indexes in memory or warm it up after a node or cluster restart because the Durable Memory is tightly coupled with persistence and treats it as a secondary memory tier. This implies that if a subset of data or an index is missing in RAM, the Durable Memory will take it from the disk.</p> 
          <p><a href="https://apacheignite-cpp.readme.io/docs/ignite-persistence" target="_self">Read More</a></p> 
         </div> 
         <div class="magic-block-api-header"> 
          <h1 class="header-scroll is-api-header"><span id="acid-compliance"></span> 
           <div id="section-acid-compliance" class="anchor waypoint"></div>ACID Compliance<a href="#section-acid-compliance" class="fa fa-anchor"></a> </h1> 
         </div> 
         <div class="magic-block-textarea">
          <p>Data stored in Ignite is ACID-compliant both in memory and on disk, making Ignite a <strong>strongly consistent</strong> system. Ignite transactions work across the network and can span multiple servers.</p> 
          <p><a href="https://apacheignite-cpp.readme.io/docs/transactions" target="_self">Read More</a></p> 
         </div> 
         <div class="magic-block-api-header"> 
          <h1 class="header-scroll is-api-header"><span id="complete-sql-support"></span> 
           <div id="section-complete-sql-support" class="anchor waypoint"></div>Complete SQL Support<a href="#section-complete-sql-support" class="fa fa-anchor"></a> </h1> 
         </div> 
         <div class="magic-block-textarea">
          <p>Ignite provides full support for SQL, DDL and DML, allowing users to interact with Ignite using pure SQL without writing any code. This means that users can create tables and indexes as well as insert, update, and query data using only SQL. Having such complete SQL support makes Ignite a one-of-a-kind <strong>distributed SQL database</strong>.</p> 
          <p><a href="https://apacheignite-cpp.readme.io/docs/sql-database" target="_self">Read More</a></p> 
         </div> 
         <div class="magic-block-api-header"> 
          <h1 class="header-scroll is-api-header"><span id="key-value"></span> 
           <div id="section-key-value" class="anchor waypoint"></div>Key-Value<a href="#section-key-value" class="fa fa-anchor"></a> </h1> 
         </div> 
         <div class="magic-block-textarea">
          <p>The in-memory data grid component in Ignite is a fully transactional <strong>distributed key-value store</strong> that can scale horizontally across 100s of servers in the cluster. When persistence is enabled, Ignite can also store more data than fits in memory and survive full cluster restarts.</p> 
          <p><a href="https://apacheignite-cpp.readme.io/docs/data-grid" target="_self">Read More</a></p> 
         </div> 
         <div class="magic-block-api-header"> 
          <h1 class="header-scroll is-api-header"><span id="collocated-processing"></span> 
           <div id="section-collocated-processing" class="anchor waypoint"></div>Collocated Processing<a href="#section-collocated-processing" class="fa fa-anchor"></a> </h1> 
         </div> 
         <div class="magic-block-textarea">
          <p>Most traditional databases work in a client-server fashion, meaning that data must be brought to the client side for processing. This approach requires lots of data movement from servers to clients and generally does not scale. Ignite, on the other hand, allows for sending light-weight computations to the data, i.e. <strong>collocating</strong> computations with data. As a result, Ignite scales better and minimizes data movement.</p> 
          <p><a href="https://apacheignite.readme.io/docs/collocate-compute-and-data" target="_self">Read More</a></p> 
         </div> 
         <div class="magic-block-api-header"> 
          <h1 class="header-scroll is-api-header"><span id="scalability-and-durability"></span> 
           <div id="section-scalability-and-durability" class="anchor waypoint"></div>Scalability and Durability<a href="#section-scalability-and-durability" class="fa fa-anchor"></a> </h1> 
         </div> 
         <div class="magic-block-textarea">
          <p>Ignite is an elastic, horizontally scalable distributed system that supports adding and removing cluster nodes on demand. Ignite also allows for storing multiple copies of the data, making it resilient to partial cluster failures. If the persistence is enabled, then data stored in Ignite will also survive full cluster failures. Cluster restarts in Ignite can be very fast, as the data becomes operational instantaneously directly from disk. As a result, the data does not need to be preloaded in-memory to begin processing, and Ignite caches will lazily warm up resuming the in memory performance.</p> 
          <p><a href="https://apacheignite-cpp.readme.io/docs/cluster" target="_self">Read More</a></p> 
         </div> 
         <div class="magic-block-api-header"> 
          <h1 class="header-scroll is-api-header"><span id="ignite-and-ignite-с"></span> 
           <div id="section-ignite-and-ignite-с" class="anchor waypoint"></div>Ignite and Ignite С++<a href="#section-ignite-and-ignite-с" class="fa fa-anchor"></a> </h1> 
         </div> 
         <div class="magic-block-textarea">
          <ul> 
           <li>Ignite C++ is built on top of Ignite</li>
           <li>Ignite С++ starts the JVM in the same process and communicates with it via JNI</li>
           <li>.NET, C++ and Java nodes can join the same cluster, use the same caches, and interoperate using common binary protocol</li>
           <li>Java compute jobs can execute on any node (Java, .NET, C++)</li>
          </ul> 
         </div>
